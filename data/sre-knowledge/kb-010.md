# Monitoring and diagnosing health check endpoints

## Overview
Each microservice exposes the following health check endpoints:
- GET /health - Basic liveness check
- GET /health/ready - Readiness check including dependent services (Cosmos DB)

## Monitoring Methods
### Liveness Check
Checks whether the application process can respond. If HTTP 200 is returned, the process is healthy.
```bash
curl http://localhost:5001/health  # CatalogService
curl http://localhost:5002/health  # OrderService
curl http://localhost:5003/health  # NotificationService
```

### Readiness Check
Checks whether the application can process requests. Includes connectivity to Cosmos DB.
```bash
curl http://localhost:5001/health/ready  # CatalogService
curl http://localhost:5002/health/ready  # OrderService
curl http://localhost:5003/health/ready  # NotificationService
```

## Container Apps Probe Settings
- Startup Probe: /health, initial delay 5 seconds, interval 10 seconds, failure threshold 3
- Liveness Probe: /health, interval 30 seconds, failure threshold 3
- Readiness Probe: /health/ready, interval 15 seconds, failure threshold 3

## Incident Response
- Liveness check failure: The container is restarted automatically
- Readiness check failure: Traffic is no longer routed. Check the Cosmos DB connection
- Both checks fail: Check container logs and identify the cause of the startup error
