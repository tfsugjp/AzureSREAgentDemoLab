# NotificationService notification delivery delays or missing notifications

## Symptoms
Notifications created by NotificationService cannot be retrieved with GET /api/notifications?userId={userId}, or notification delivery is delayed.

## Possible Causes
1. Delay caused by the Cosmos DB indexing policy (Eventual Consistency)
2. Incorrect partition key (UserId)
3. Temporary throttling in Cosmos DB Serverless mode
4. Issue with notification IsRead filtering

## Verification Steps
1. Check the response when creating a notification with POST /api/notifications (whether 201 Created is returned)
2. Check Cosmos DB metrics for 429 (Too Many Requests) errors
3. Verify that the notification userId is correct
4. Check the indexing policy for the Cosmos DB 'notifications' container

## Mitigation
- Change the Cosmos DB consistency level from Session to Strong (if read-after-write consistency is required)
- When throttling occurs: Verify retry logic and implement appropriate backoff
- Verify the partition key: Use the same UserId for notification creation and retrieval

## API Endpoints
- POST /api/notifications - Create notification
- GET /api/notifications?userId={userId} - Retrieve notifications by user
- PUT /api/notifications/{id}/read - Mark as read
