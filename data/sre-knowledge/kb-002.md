# CatalogService HTTP 500 error caused by long query strings

## Symptoms
On the CatalogService product search endpoint GET /api/products/search, an HTTP 500 Internal Server Error occurs when the q query parameter exceeds 100 characters.

## Root Cause
The SearchAsync method in ProductSearchService.cs contains a String.Substring boundary-checking bug in the logic that handles queries longer than 100 characters. Specifically:
1. query.Substring(0, 100) extracts the first 100 characters
2. query.Substring(100, 100) then attempts to extract the next 100 characters
3. If the query is shorter than 200 characters, the second Substring throws ArgumentOutOfRangeException

## Impact
- The CatalogService /api/products/search endpoint
- Requests where the query string is longer than 100 characters and shorter than 200 characters

## Mitigation
1. Immediate action: Limit query string length at the API Gateway or reverse proxy
2. Permanent fix: Correct the Substring boundary checks and replace them with appropriate string truncation logic
3. Recommended: Validate query strings at the endpoint layer and reject excessively long queries with 400 Bad Request

## Reproduction Command
```bash
curl -s -o /dev/null -w '%{http_code}' "http://localhost:5001/api/products/search?q=$(python3 -c 'print("a"*150)')"
```

## Related Logs
The stack trace includes System.ArgumentOutOfRangeException.
