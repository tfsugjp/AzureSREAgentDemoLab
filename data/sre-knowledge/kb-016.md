# Troubleshooting CatalogService inventory count inconsistencies

## Symptoms
The inventory count returned by CatalogService inventory information (GET /api/inventory/{productId}) differs from the expected actual value. The reserved count (reservedQuantity) does not match the actual number of orders.

## Possible Causes
1. Inventory synchronization between OrderService and CatalogService is manual, causing inconsistencies
2. Optimistic concurrency conflicts caused by concurrent updates
3. Mismatch between seed data initial values and actual order state
4. Temporary inconsistency caused by Cosmos DB eventual consistency

## Verification Steps
1. Retrieve current inventory information with GET /api/inventory/{productId}
2. Check orders for that product with GET /api/orders
3. Check the document directly in the Cosmos DB inventory container
4. Compare reservedQuantity with the actual number of Pending/Processing orders

## Mitigation
- Manual inventory correction: Update to the correct value with PUT /api/inventory/{productId}
- Reconciliation: Scan all orders and recalculate reservedQuantity
- Concurrent update mitigation: Consider implementing optimistic concurrency control using Cosmos DB ETag

## Seed Data Inventory
- Initial inventory of 10 to 500 units for each of 20 products
- Each product has a reserved count (1 to 30) and reorder threshold (5 to 50) configured
