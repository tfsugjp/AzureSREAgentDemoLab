# Cosmos DB partition key design and query performance

## Partition Key Design

### Partition Keys for Each Container
| Container | Partition Key | Description |
|----------|---------------------|------|
| categories | /id | Category ID |
| products | /categoryId | Partitioned by category ID |
| inventory | /productId | Partitioned by product ID |
| orders | /userId | Partitioned by user ID |
| notifications | /userId | Partitioned by user ID |

### Query Performance Considerations
- Queries that specify the partition key run on a single partition and are efficient
- Queries that do not specify the partition key become cross-partition queries and are inefficient because they scan all partitions
- Searches on the products container (GET /api/products/search) use cross-partition queries

### Serverless Mode Constraints
- Maximum 5,000 RU per request
- Maximum storage of 1 TB
- SLA differs from Standard accounts
- Cold starts may occur after long idle periods

### Performance Optimization
1. Include the partition key in queries
2. Session consistency is sufficient for read-heavy workloads
3. Avoid SELECT for unnecessary properties and project only the required properties
4. Monitor RU consumption with Cosmos DB metrics
