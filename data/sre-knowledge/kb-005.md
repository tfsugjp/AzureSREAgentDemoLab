# Container Apps scaling issues

## Symptoms
Container Apps instances remain scaled down to 0 and do not recover, or scale-out is delayed. Cold starts cause the first request to have significantly increased response latency.

## Possible Causes
1. In non-production environments, minReplicas is set to 0, so instances scale to 0 when idle
2. Pulling the container image takes time (from ACR)
3. The .NET application loads Cosmos DB seed data during startup
4. The health check timeout setting is too short

## Verification Steps
1. Check the Container Apps replica count in the Azure Portal
2. Check Container Apps logs for startup events
3. Check the image pull status from ACR
4. Check responses from health check endpoints (/health, /health/ready)

## Mitigation
- Production environment: Set environmentType to 'prod' so minReplicas becomes 1 and cold starts are avoided
- Non-production environment: Manually change minReplicas to 1 if needed
- Speed up ACR pulls: Change the ACR SKU from Basic to Standard
- Optimize seeding: Reduce initialization time in StartupInitializationService

## Related Configuration
- Bicep: minReplicas / maxReplicas variables in infra/main.bicep
- Bicep: Health probe settings in infra/modules/container-app.bicep
