# Detecting and mitigating thread pool starvation

## Overview
When synchronous blocking with Thread.Sleep occurs in the CatalogService Premium search path, it can cause .NET thread pool starvation.

## Detection Methods

### Application Insights Metrics
- Average request response time spikes (normally < 100ms → several seconds or more)
- Concurrent request count decreases (new requests cannot be accepted due to thread starvation)
- Dependency call timeouts increase

### Log-Based Detection
- Frequent occurrence of the "Performing premium product search with enhanced filtering" log
- Available thread count decreases in ThreadPool.GetAvailableThreads

### Synthetic Monitoring
- Response time for the /health health check increases
- /health/ready times out

## Immediate Mitigation
1. Temporarily block traffic to the premium search path (API Gateway rule)
2. Manually scale out the number of Container Apps instances
3. Restart the CatalogService container (not a root-cause fix)

## Permanent Mitigation
1. Change Thread.Sleep(100) to await Task.Delay(100)
2. Replace bulk retrieval of all products with direct filtering in the Cosmos DB query
3. Apply asynchronous patterns consistently
